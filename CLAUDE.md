# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

latoile is a "context bridge": from a single entry-point Jira key it recursively walks the Jira relationship graph (parent, subtasks, siblings, links, key mentions in text), enriches each issue with GitLab data (MRs, branches, commits), and emits both a renderable graph (`{ nodes, edges }`) and a normalized LLM context payload. Auth piggybacks on the locally logged-in `acli` (Atlassian) and `glab` (GitLab) CLIs: Jira goes through the `acli` binary, GitLab through direct `fetch()` calls with the token read at runtime from glab's config file (`src/collector/gitlab-http.ts`). No tokens in the repo; a gitignored `.env` is auto-loaded by `src/config.ts`.

## Commands

Use **yarn** (not npm) for all package/script commands in this repo.

```bash
yarn build          # compile backend (tsconfig.json → dist/) and frontend (tsconfig.web.json → public/app.js)
yarn typecheck      # both tsconfigs, --noEmit
yarn start          # builds backend then serves Express API + UI at http://localhost:3000
yarn graph JIRA-123 --view full   # CLI fetch, no UI (dist/src/cli.js)
yarn test           # compiles backend then runs node --test dist/test/*.test.js
```

Run a single test file (tests run against compiled output, so build first):

```bash
yarn build:server && node --test dist/test/traversal.test.js
```

Lint with `yarn lint` — ESLint flat config (`eslint.config.js`) with the latest JS + typescript-eslint recommended rules; `no-explicit-any` is an error per repo convention. Note: `typescript` is pinned to 5.9 because typescript-eslint cannot parse with the TypeScript 7 native preview. Tests use the Node built-in test runner (`node:test`), no test framework dependency.

## Architecture

Strict TypeScript, ESM (`"type": "module"` — source imports use `.js` extensions). Two compile targets: `tsconfig.json` builds `src/` + `test/` to `dist/` for Node; `tsconfig.web.json` builds `src/web/app.ts` to `public/app.js` for the browser. The repo convention is no `any`/`unknown` in the codebase.

Data flow: `src/pipeline.ts` (`buildContextGraph`) wires everything —

- `src/collector/runner.ts` — promisified `execFile` wrapper (argument arrays, no shell) plus a runner adding delay/retry/timeout, configured via `LATOILE_*` env vars read in `src/config.ts`.
- `src/collector/acli.ts` — Jira client shelling out to `acli`; also parses the dev-status summary (`customfield_10000`) into `hasGitlabData` so the traversal can skip GitLab lookups for issues Jira says have none.
- `src/collector/gitlab-http.ts` — the GitLab client the pipeline actually uses: `fetch()` with the token read from glab's config (or `LATOILE_GITLAB_TOKEN`); ~15× faster than spawning `glab`. `src/collector/glab.ts` is the older process-spawning client and still owns the shared normalizers (`normalizeMergeRequest`, `normalizeCommit`, `pooledMap`). Project resolution from `groups` is resilient: `fetchGroupProjects` retries each page once (these gitlab.com group endpoints time out intermittently) and returns a `complete` flag; `resolveProjects` only caches a fully-complete scan, so a transient failure is returned best-effort for that call but never poisons `_cachedProjects` for the process lifetime (`lastResolutionDegraded` records the outcome). This was the real cause of "MRs silently empty for a ticket that clearly has them".
- `src/collector/traversal.ts` — breadth-first traversal from the entry key with visited-set and `maxDepth`/`maxNodes` limits; keys beyond the limits become unresolved placeholder nodes. Relations carry `strength: 'strong' | 'weak'` (weak = text mention).
- `src/model/graph.ts` — `buildGraph` (visualization payload) and `buildContext` (LLM payload) from the traversal result. `EDGE_SCHEMA` declares the valid source/target node types per edge type. Branches and commits are not graph nodes — they're folded into the MR node (`sourceBranch`, `commitCount`, `commits`) and shown in the UI details panel; the context payload still lists commits per issue. `buildContext` also emits a `traversal` block (`ContextTraversalInfo`) surfacing `stats` so a consumer can tell a genuine empty neighborhood from a budget-truncated one: `node_cap_hit` and `depth_limit_hit` are kept separate (raise `maxNodes` vs. raise `maxDepth`). The KG-served path (`knowledge-graph.ts` `storedContext`) populates the node-count fields for parity but omits `depth_reached`/`depth_limit_hit` — the single variable-length stored query can't report per-node depth, and an optimistic value would give a false "depth complete".
- `src/api/server.ts` — Express backend; `GET /api/graph/:key` runs the pipeline live on each request (`?view=context|full`, `?maxDepth=`, `?maxNodes=`; with `Accept: text/event-stream` it streams progress logs then the result). `GET /api/search?q=` proxies a JQL text search for the UI autocomplete. Serves `public/` as the frontend.
- `src/cli.ts` — same pipeline as a one-shot command (`latoile` bin).
- `src/cache/` — SQLite fetch cache (`node:sqlite`, no native dep; requires Node ≥ 22.13). `store.ts` defines the storage-agnostic `CacheStore` + `SqliteCacheStore` (default file `~/.latoile/cache.db`); `cached-clients.ts` wraps the traversal's `IssueSource`/`GitlabSource` so clients and traversal stay cache-unaware. Config: `LATOILE_CACHE=off` disables, `LATOILE_CACHE_PATH`, `LATOILE_CACHE_TTL_MIN` (default 15). `?refresh=1` on the API (or `refresh` pipeline option) bypasses reads but still writes. Null Jira results are never cached.
- `src/mcp/` — stdio MCP server (`latoile-mcp` bin, `yarn mcp`). `server.ts` wires tools to the transport and owns startup/shutdown; `handlers.ts` has the transport-free tool handlers (unit-tested directly); `lifecycle.ts` tracks in-flight calls and the shared Neo4j read handle so shutdown can drain and disconnect; `tool-result.ts` is the shared result shape. `yarn smoke` (`scripts/smoke-mcp.mjs`) exercises the real server over stdio: handshake, tools/list, an optional tool call, and exit-on-EOF. Eight tools: `get_context` (full traversal), `get_context_from_mr` (GitLab MR URL entry point — `src/collector/mr-entry.ts` parses the URL and extracts the Jira key from source branch → title → description), `search_issues` (JQL text search, shared logic in `src/collector/search.ts`), `get_issue` (single fetch, no traversal), plus five offline knowledge-graph tools (`find_connection`, `known_context`, `person_activity`, `project_activity`, `graph_stats` — canned Cypher via `src/sink/knowledge-graph.ts`, require `LATOILE_NEO4J_URI`). `get_context` also accepts `maxAgeSeconds` (two-tier incremental refresh): a fully fresh stored neighborhood is served straight from the knowledge graph (`source: 'knowledge_graph'`); otherwise the traversal runs with KG-backed client decorators (`src/sink/kg-clients.ts`) that serve each within-budget issue from the graph and live-fetch only the stale frontier (`source: 'partial'`, `graphServedIssues`). Graph-served issues carry `provenance: 'knowledge_graph'` and are excluded from sink re-ingestion so `last_seen` always means "last verified live". All tools return `structuredContent` with an output schema; pipeline logs are forwarded as MCP logging notifications and, when the client sends a `progressToken`, as progress notifications. Handlers (`getContextTool`, `searchIssuesTool`, `getIssueTool`) are exported separately for tests (`test/mcp.test.ts`). Progress logs go to stderr (stdout is the JSON-RPC channel — never console.log there).
- `src/sink/` — knowledge-graph persistence (PLAN-NEO4J.md). `graph-sink.ts` defines the `GraphSink` interface; `neo4j-sink.ts` implements it with batched Cypher MERGE upserts (issues incl. unresolved placeholders, MRs, commits, people, docs; `first_seen`/`last_seen` timestamps; sibling relations deliberately not persisted). GitLab projects are first-class (`:Project {path, gitlabId}`, `IN_PROJECT` edges) because one work item spans several repos; the context payloads surface this as `repositories` per item and per context. The pipeline calls `sink.ingest(traversal)` fire-safe after each run. Enabled only when `LATOILE_NEO4J_URI` is set (`LATOILE_NEO4J_USER`/`LATOILE_NEO4J_PASSWORD`, `LATOILE_NEO4J=off` kill switch); `neo4j-driver` is lazy-imported so it never loads unconfigured. Local DB: `docker-compose.yml` (container `latoile-neo4j`, data volume `D:\DockerVolumes\neo4j`).
- `src/collector/orbit.ts` — optional (`LATOILE_ORBIT=1`) enrichment shelling out to the `orbit` CLI against a local [GitLab Orbit](https://docs.gitlab.com/orbit/) DuckDB graph (see PLAN-ORBIT.md). Matches a GitLab repo path to Orbit's manifest by last-path-segment (lowercased) — Orbit's `project_id` is a synthetic hash, not GitLab's, and is kept as a string throughout (exceeds `Number.MAX_SAFE_INTEGER`). Attaches a `code` neighborhood per repo an issue's MRs touched (needs `LATOILE_GITLAB_FETCH_FILES=1` for changed-file paths); the block is branch-aware — Orbit reflects whatever branch is locally checked out, not the MR's branch, so it distinguishes not-indexed / indexed-but-no-match (branch drift) / matched rather than presenting stale code as authoritative.
- `src/types.ts` — shared domain types (node/edge types, context payload shapes).

Dependency injection for tests: `createClients` in `pipeline.ts` and the runner's `exec` option exist so tests can stub CLI execution instead of invoking real `acli`/`glab` (see `test/collector.test.ts`, `test/traversal.test.ts`).

## Gotchas

- `acli jira workitem view` takes the key positionally (`view KEY-123`), not via `--key`. The default field set excludes `issuelinks`/`subtasks`/`parent`/`comment`, so the client passes `--fields '*all'`.
- `glab mr list` only works inside a git repo with a GitLab remote, and group-level `?search=` requests time out on gitlab.com — use project-scoped API queries (`projects/<url-encoded-path>/merge_requests?search=KEY&in=title`), which is what `gitlab-http.ts` does.
- Jira ↔ GitLab correlation is done by the "GitLab for Jira Cloud" plugin; only its summary is readable from the issue payload (`fields.customfield_10000`). The detail dev-status API isn't reachable through acli, so MRs are found by searching titles/branches for the Jira key (team convention: `fix/PV2-XXXXX-...`).
- GitLab enrichment needs `LATOILE_GITLAB_GROUPS` or `LATOILE_GITLAB_PROJECTS` set (usually via the gitignored `.env`); with neither, it silently returns no MRs (a warning is logged).
- GitLab request volume is O(issues × projects): `fetchForKey` searches *every* resolved project per Jira key (`gitlab-http.ts`), and the traversal repeats that for each resolved issue — so higher `maxDepth`/`maxNodes` and large groups can trip gitlab.com's rate limit. Mitigations already in place: all requests funnel through `apiGet`, which on HTTP 429 backs off honoring `Retry-After`/`RateLimit-Reset` (capped, `LATOILE_GITLAB_MAX_RETRIES`, default 4); if 429s persist, lower `LATOILE_GITLAB_CONCURRENCY` (default 8) or narrow the configured groups/`activeDays`.
- The UI defaults to dark and has a light/dark toggle; both palettes come from the company design tokens as CSS variables in `public/styles.css`. The Cytoscape colors are read from those variables at render time (`getThemeColor` in `src/web/app.ts`).
- `acli` calls take ~5s each (process spawn); that's the pipeline bottleneck. See `PLAN.md` for pending work.

## Domain conventions

Jira ↔ GitLab linking relies on the Jira key appearing in branch names (`feature/JIRA-123-*`), MR titles/descriptions, and commit messages. Node types: `jira`, `merge_request`, `doc` (branches/commits ride on the MR node, see above). Jira comments are likewise not nodes: they ride on the issue (`comments` on the normalized issue and context item, capped at the 20 most recent in the context payload and the Neo4j `:Issue` node, where they're stored as JSON strings — `src/sink/comment-codec.ts`); comment text feeds mention extraction, so a key cited only in a comment still yields a weak `mention` edge. Edge types: `parent`, `subtask`, `sibling`, `link`, `mention` (Jira ↔ Jira); `has_mr` (Jira → MR); `documented_by` (Jira → doc). Source/target constraints live in `EDGE_SCHEMA` (`src/model/graph.ts`); every edge has `strength: 'strong' | 'weak'` (weak = text mention only). `sibling` edges are emitted once per pair (sorted key order) and only when the shared parent is not itself a node in the graph — when it is, parent/subtask edges already convey the relationship, and a full clique grows quadratically with family size.
